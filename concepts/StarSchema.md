# Star Schema

## My understanding
A star schema is a data warehouse design where one central **fact table** holds numeric metrics and foreign keys, surrounded by **dimension tables** that hold the descriptive context. It looks like a star when drawn. It's the opposite of a normalized OLTP schema — it's intentionally denormalized for fast reads.

## Why it matters
OLTP schemas need many JOINs to answer analytical questions. A star schema pre-joins and pre-aggregates the data so dashboard queries become simple. Queries that needed 4 JOINs on the source now need 1–2.

## Example
```
          dim_date          dim_agent
         (date_key)        (agent_key)
              \                 /
               \               /
            fact_ticket_daily
         (date_key, agent_key, status,
          priority, tickets_created,
          tickets_resolved)
```

```sql
-- One clean query instead of joining 3 OLTP tables
SELECT f.date_key, a.agent_name, f.tickets_created, f.tickets_resolved
FROM   fact_ticket_daily f
JOIN   dim_agent         a ON a.agent_key = f.agent_key;
```

# Fact Table vs Dimension Table

## My understanding
- **Fact table** — stores measurable events (one row per grain). Columns are numbers (counts, sums, averages) plus foreign keys to dimensions. It's the "what happened" table.
- **Dimension table** — stores descriptive attributes about entities (agents, dates, statuses). It's the "who/when/what" context.

## Why it matters
Separating metrics from context keeps fact tables narrow and fast to scan. Dimension tables are small and rarely change, so they're cheap to JOIN.

## Example
```sql
-- Fact table: metrics at a specific grain
CREATE TABLE fact_ticket_daily (
    date_key          NUMBER NOT NULL,
    agent_key         NUMBER NOT NULL REFERENCES dim_agent(agent_key),
    status            VARCHAR2(20) NOT NULL,
    priority          VARCHAR2(10) NOT NULL,
    tickets_created   NUMBER DEFAULT 0,
    tickets_resolved  NUMBER DEFAULT 0
);

-- Dimension table: context about agents
CREATE TABLE dim_agent (
    agent_key   NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    agent_name  VARCHAR2(100) NOT NULL,
    team        VARCHAR2(50)  NOT NULL
);
```

# Surrogate Key

## My understanding
A surrogate key is an artificial primary key (usually an auto-incremented number) added to dimension tables. It's different from the source system's ID (the natural key). The fact table references the surrogate key, not the source ID.

## Why it matters
It decouples the data warehouse from the source system. If the source system renumbers its IDs, or if the same entity comes from multiple sources, the DW surrogate key stays stable.

## Example
```sql
-- dim_agent.agent_key is the surrogate (auto-generated by DW)
-- The source system just uses integers 1, 2, 3... in tickets.assigned_to
-- They happen to match here, but in production they often don't
INSERT INTO dim_agent (agent_name, team) VALUES ('Sara Lopez', 'Tier-1 Support');
-- agent_key = 1 is assigned by the DW, independent of any source ID
```
