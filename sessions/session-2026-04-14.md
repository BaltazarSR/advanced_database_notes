# Session – 2026-04-14

## Topics covered
- What a database index is and how it works internally 
- Full table scan vs index range scan — how the optimizer chooses
- Cardinality: why low-cardinality columns (like `status` or `site_id`) often don't benefit from an index
- Composite indexes and the leading column rule

## What I understood
- An index is a separate structure the DB maintains so it can jump to matching rows without scanning everything — the core trade-off is faster reads at the cost of slower writes and extra storage.
- The optimizer decides whether to use an index based on selectivity. If a query would return a large percentage of the table, a full scan is cheaper and the index is ignored even if it exists.
- Low cardinality (few distinct values) = low selectivity = index often ignored. High cardinality (many distinct values like `patient_id`) = high selectivity = index is very effective.
- Composite index column order matters: the index is only useful if the leading column appears in the WHERE clause. Querying only on a trailing column means Oracle cannot use the composite index.
- Wrapping a column in a function (`TO_CHAR(patient_id)`) silently breaks index use because the index stores raw values, not computed results. The query still returns correct data but does a full table scan.
