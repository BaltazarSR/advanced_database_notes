## Topics covered
- Transaction control: COMMIT, ROLLBACK, SAVEPOINT
- How Oracle handles partial failures and why atomicity matters
- Stored procedures: structure, parameters, error handling, COMMIT/ROLLBACK inside PL/SQL
- Functions vs procedures: return values, side effects, and where each can be used

## What I understood
- A transaction is all-or-nothing. COMMIT makes changes permanent; ROLLBACK undoes everything back to the last commit. SAVEPOINT lets you mark a checkpoint so you can partially undo without losing all your work.
- The optimizer doesn't decide here — transaction boundaries are controlled by the developer. If you forget to COMMIT, changes are only visible to your session.
- A stored procedure packages multi-step logic in the database. It can own its transaction (COMMIT/ROLLBACK inside the EXCEPTION block), so the caller always gets a clean outcome even on failure.
- The procedure vs function distinction is strict in Oracle: procedures do things (DML, COMMIT), functions return values and cannot have side effects. Functions can appear in SELECT; procedures cannot.
- A procedure that calls COMMIT internally will commit the caller's open transaction too — this is a hidden coupling that can break larger workflows.
