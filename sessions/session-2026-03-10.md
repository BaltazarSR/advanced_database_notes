## What I understood
- Set operations work like mathematical sets, combining query results vertically rather than horizontally like JOINs do.
- The key rule is that all queries must have the same number of columns with compatible data types—otherwise the operation fails.
- UNION removes duplicates automatically, which is great for clean reporting but has a performance cost. UNION ALL keeps everything and runs faster.
- INTERSECT is perfect for finding overlaps—like customers who appear in multiple time periods or meet multiple criteria.
- MINUS/EXCEPT is incredibly useful for finding gaps or exclusions, like identifying users who registered but never converted.
- These operations are simpler than JOINs when you don't have a clear relationship between tables but still need to combine their results.

## What is still confusing
- When UNION vs UNION ALL performs significantly differently in large datasets—at what scale does the duplicate-checking overhead really matter?
- Whether you can combine different set operations in one query (like chaining UNION with INTERSECT) and how the order of operations affects the result
